# Terraform Beginner Guide — How This Project Was Built

## Why this guide exists

This project is not just meant to deploy an S3 static website.

It is meant to answer the beginner question:

> If someone says "build something with Terraform," where do I even start?

This guide breaks down the thinking process behind the project structure.

---

## 1. Terraform does not require a magic bootstrap project

There is no required "Terraform bootstrap download" for every project.

At the most basic level, a Terraform project is just a folder with `.tf` files.

Example:

```text
my-terraform-project/
├── main.tf
├── variables.tf
├── outputs.tf
└── providers.tf
```

Terraform reads all `.tf` files in the folder together.

That means Terraform does not care whether everything is in one file or split across multiple files.

This would technically work:

```text
main.tf
```

with everything inside it.

But teams usually split Terraform into multiple files to make it easier to read and maintain.

---

## 2. Where did these files come from?

The files in this project are a common Terraform convention.

They are not automatically generated by Terraform.

I created them manually based on common Terraform project patterns.

Common Terraform files:

| File | Purpose |
|---|---|
| `providers.tf` | Defines Terraform version and provider, such as AWS |
| `variables.tf` | Defines inputs the project accepts |
| `terraform.tfvars` | Provides actual values for variables |
| `locals.tf` | Defines reusable calculated values |
| `main.tf` | Defines the infrastructure resources |
| `outputs.tf` | Prints useful values after deployment |
| `.gitignore` | Prevents sensitive/local Terraform files from being committed |

## 3. Do I have to make these from scratch every time?

Not always.

### Personal / learning projects

You can start from scratch or copy a previous small project as a template.

For a tiny project, this is enough:

```text
providers.tf
main.tf
variables.tf
outputs.tf
```

### Company projects

At work, you usually do not start from nothing.

Companies often already have:

- repo templates
- Terraform modules
- CI/CD pipeline files
- naming conventions
- provider configuration
- backend/state configuration
- security scanning
- environment folders
- reusable modules

## 4. Why did I run this command?

```bash
cp terraform.tfvars.example terraform.tfvars
```

This copies the example variable values into a real local file.

### Why not just use `terraform.tfvars.example` directly?

Because `.example` files are meant to be safe templates committed to GitHub.

The real `terraform.tfvars` file often contains environment-specific or sensitive values.

For that reason, `terraform.tfvars` is usually listed in `.gitignore` and not committed.

### Simple explanation

```text
terraform.tfvars.example = template for humans
terraform.tfvars         = real values Terraform uses locally
```

## 5. How variables work

In `variables.tf`, we define what inputs Terraform expects.

Example:

```hcl
variable "environment" {
  description = "Environment name, such as dev, qa, or prod"
  type        = string
}
```

This says:

> Terraform expects a variable named `environment`, and it should be a string.

In `terraform.tfvars`, we provide the actual value:

```hcl
environment = "dev"
```

Then Terraform can use it like this:

```hcl
bucket = "${var.project}-${var.environment}-${var.owner}"
```

---

## 6. What are locals?

Locals are reusable calculated values.

Example:

```hcl
locals {
  bucket_name = lower("${var.project}-${var.environment}-${var.owner}")
}
```

Instead of rebuilding the bucket name everywhere, we define it once and reuse it:

```hcl
bucket = local.bucket_name
```

### Variables vs Locals

| Concept | Meaning |
|---|---|
| Variable | Input from outside the Terraform code |
| Local | Value calculated inside the Terraform code |

---

## 7. What are providers?

A provider is the plugin Terraform uses to talk to an external platform.

In this project, the provider is AWS.

```hcl
provider "aws" {
  region = var.aws_region
}
```

This means:

> Terraform should use AWS, and the AWS region comes from `var.aws_region`.

Other providers include:

- Azure
- Google Cloud
- GitHub
- Databricks
- Kubernetes

---

## 8. What are resources?

Resources are the actual things Terraform creates.

Examples:

- S3 bucket
- Lambda function
- DynamoDB table
- Step Function
- IAM policy
- CloudWatch rule

Example:

```hcl
resource "aws_s3_bucket" "website" {
  bucket = local.bucket_name
}
```

This creates an S3 bucket.

Terraform resource naming has two parts:

```hcl
resource "aws_s3_bucket" "website"
```

| Part | Meaning |
|---|---|
| `aws_s3_bucket` | Type of resource |
| `website` | Local Terraform name for this resource |

You can reference it elsewhere like this:

```hcl
aws_s3_bucket.website.id
```

---

## 9. What are outputs?

Outputs print useful values after deployment.

Example:

```hcl
output "website_url" {
  value = "http://${aws_s3_bucket_website_configuration.website.website_endpoint}"
}
```

After `terraform apply`, Terraform prints the website URL.

Outputs are useful for:

- URLs
- ARNs
- bucket names
- IDs
- role names

---

## 10. What is Terraform state?

Terraform state is how Terraform remembers what it created.

When you run:

```bash
terraform apply
```

Terraform creates a state file.

By default, that file is local:

```text
terraform.tfstate
```

The state file maps your Terraform code to real cloud resources.

Example:

```text
aws_s3_bucket.website in code
        ↓
actual bucket in AWS
```

### Important

Do not commit Terraform state to GitHub.

In real company environments, state is usually stored remotely, such as:

- S3 bucket for state storage
- DynamoDB table for state locking

## 11. What is the normal Terraform workflow?

The common Terraform workflow is:

```bash
terraform init
terraform fmt
terraform validate
terraform plan
terraform apply
```

### `terraform init`

Downloads providers and prepares the project.

### `terraform fmt`

Formats Terraform files.

### `terraform validate`

Checks whether the Terraform syntax is valid.

### `terraform plan`

Shows what Terraform will create, change, or destroy.

### `terraform apply`

Actually makes the changes.

### `terraform destroy`

Deletes resources managed by the project.

## 12. If I had to build this from scratch, what would I do?

### Step 1: Create a folder

```bash
mkdir terraform-static-site-mini-project
cd terraform-static-site-mini-project
```

### Step 2: Create basic files

```bash
touch providers.tf main.tf variables.tf outputs.tf locals.tf terraform.tfvars.example .gitignore
mkdir website
touch website/index.html website/error.html
```

### Step 3: Decide what to build

For this project:

> Create an S3 static website.

### Step 4: Find the resource types needed

For an S3 static website, we need resources like:

- `aws_s3_bucket`
- `aws_s3_bucket_website_configuration`
- `aws_s3_bucket_public_access_block`
- `aws_s3_bucket_policy`
- `aws_s3_object`

### Step 5: Add variables

Define inputs like:

- AWS region
- project name
- environment
- owner

### Step 6: Add resources in `main.tf`

Write the infrastructure resources.

### Step 7: Add outputs

Print the bucket name and website URL.

### Step 8: Run Terraform commands

```bash
terraform init
terraform plan
terraform apply
```

## 13. How this relates to company Terraform projects

In company repos, Terraform is usually more advanced.

You may see:

```text
terraform/
├── modules/
│   ├── lambda/
│   ├── step_functions/
│   ├── glue/
│   └── dynamodb/
├── environments/
│   ├── dev0.tfvars
│   ├── qa1.tfvars
│   └── prod1.tfvars
├── main.tf
├── variables.tf
├── outputs.tf
├── locals.tf
└── providers.tf
```

Instead of writing every resource directly, teams often use modules.

A module is reusable Terraform code.

Example:

```hcl
module "glue_job" {
  source = "./modules/glue"

  job_name = local.glue_job_name
}
```

This means:

> Use the reusable Glue module and pass values into it.
